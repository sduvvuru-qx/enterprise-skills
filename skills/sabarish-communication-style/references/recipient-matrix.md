# Recipient Matrix

Use this to choose tone, structure, and level of detail by audience.

## Bryan Voris

**Role lens:** department head, executive visibility, AI proposal sponsor, escalation path.

**Best use:** alignment, direction, support request, policy/IT path, department-level risk.

**What he needs first:** business or execution implication.

**Good structure:**

```text
Bryan,

Bottom line: [impact / decision point].

[One paragraph of context.]

Suggested next step: [specific path or ask].
```

**Use when:** asking for direction, clarifying approved path, elevating a blocker.

**Avoid:** raw tool details, unresolved engineering noise, broad philosophical AI discussion.

**Phrase bank:**

- `The open question may be what approved path IT wants engineering to use.`
- `I can pull together the request status and short comparison if that helps.`
- `Current understanding is that this is more of an approved-workflow question than a tool preference question.`

## Kevin Butler

**Role lens:** manager, execution triage, schedule/resource/risk decisions.

**What he needs first:** blocked vs pending, owner, recommended action.

**Good structure:**

```text
Kevin,

Bottom line: [status/recommendation].

Open item:
- [owner] - [item] - [impact/date]

My recommendation: [action].
```

**Avoid:** vague `let me know if you need updates`; mixing ownership without labels.

## Yingying Gui

**Role lens:** peer/lead engineer raising RFQ requirements-management use case.

**What she needs first:** support, practical next step, connection to approved workflow.

**Good structure:**

```text
Hi Yingying,

I agree this use case makes sense, especially for large RFQ document sets.

The main thing is to keep confidential data within the approved path. I can help compare what M365 Copilot can cover vs what may need GitHub Copilot / IT-approved engineering tooling.
```

**Avoid:** making it sound like her idea is non-compliant; tool advocacy before use-case framing.

## JP counterparts

**Role lens:** cross-region confirmation, schedule/drawing/process alignment.

**What they need first:** purpose and intended use.

**Seniority rule:** use `-san` and order Japanese names by seniority in greetings, To/Cc, and action lists when practical. If seniority is unknown, preserve original thread order or ask the user.

**Good structure:**

```text
[Name]-san,

Quick confirmation for [program/topic].

Purpose: [where this will be used]
Current status: [fact]
Open item: [specific confirmation]

Could you confirm [exact ask]?
```

**Avoid:** idioms, vague deadlines, parallel escalation without routing confirmation.

## Customer / supplier

**Role lens:** external or semi-external record. Must be forward-safe.

**What they need first:** direct answer and confirmed facts.

**Good structure:**

```text
[Greeting]

[Direct answer.]

Confirmed:
- [fact]

Pending:
- [owner / confirmation]

Next step:
- [action]
```

**Avoid:** internal tool/process detail, blame, unapproved commitments, speculation.

## Wide CC internal thread

**Role lens:** many readers, unknown forwarding path.

**What they need first:** non-controversial framing and clear next step.

**Good structure:**

```text
Hi all,

Bottom line: [neutral bottom line].

For next step, I suggest [action] so we can [reason].
```

**Avoid:** jokes, side comments, criticism, uncertain claims.

## Recipient checklist

Before drafting, answer:

- Is this person a decision maker, reviewer, contributor, or FYI?
- Should the message ask, inform, align, or escalate?
- What would this person care about first?
- Is the current thread already enough context?
- Would a shorter Teams message be better than an email?
